Data Science Workflow

Posted on 09 December 2016 in Data Science

In this post, I'm just going to explain how to approach a data science problem step-by-step.

1. Define the problem

The first step in any data science problem is to understand what you're trying to figure out. Maybe you're looking at handwritten images of numbers and you want the machine to learn what those number actually are. Maybe you want to see who survived the Titanic. Maybe you want to calculate income based on demographic data or determine if someone will relapse given gene expression data. Defining the problem is important because this how you will determine what data you want to collect.

2. Collect data

Of course, in any data science problem, you're going to need data. In this step, you want to collect any data that you think may be relevant in solving your problem. Maybe you're researching a well-defined problem and others have already collected data, meaning you can just download data, either from a database or as a CSV. This is where SQL will be very useful. SQL is a language that allows you to access information from different databases. The common database servers are MySQL, Microsoft SQL and PostreSQL, but SQL will allow you to access information from most (all?) relational databases and pull the information you need. If you're working with structured data, the data will come in an organized fashion, usually in the form of a table. Each row of the table will be a single observation and each column will be a feature or characteristic; thus each cell in the table is a feature for a specific observation. For example if you're looking at Titanic passenger information, each row would represent a unique passenger and each column would represent a feature about the passengers. So you may have columns for Name, Age, Sex, Passenger Class, etc.

3. Clean the data

Now that you know what you're working with, you'll have to clean the data. This is a crucial step. This is where you find and hopefully fix any missing or incorrect data. Data will rarely if ever be clean. Going back to the Titanic example, there will be some observations where the Age or Sex or Class is missing for some passengers. Or (possibly) even worse, the data is incorrect (maybe someone's Age is listed as 324 years old or someone who was actualy 27 is listed as 72). There really isn't an easy way to identify and fix the incorrect data unless there are some really outrageous flaws (like 324 years old). For missing data, it's up to you to figure what you want to do with it. You can get rid of observations with missing data, but then you're losing valuable data. You can fill the missing values with the mean of the column, which could be a good approach, but you're losing the integrity of the data. Another option is to randomly assign a value taken from the distribution of the feature, which again loses some integreity. There really is not simple solution for cleaning data, but the general rule here is that you want to keep as much of the data as possible without sacrificing too much integrity. This is arguably the most difficult step of the process.

4. Feature selection

Once your data is nice and clean, you need to select which features you will use to build your predictive model. Going back to the Titanic example again, if some of the features you cleaned are Age, Sex, Name and Passenger Class, maybe you want to drop the Name feature because you don't think it has much predictive value for determining who survived the shipwreck. Feature selection is also a very difficult process. Why not keep every feature? Well, what's the point? If you know Name won't have any predictive value, why try to use it to make a prediction? Having too many features will also lead to overfitting -- a problem in which your predictive model is too sensitive to the data you used to build it which leads to poor generalizability. If you find a new set of passengers on the Titanic that you didn't know about when you were building your model, you want your model to be able to predict survival for that set as well. Another problem you need to consider is multicollinearity. If 2 features measure the same thing or similar things (they have high correlation with each other), putting both features in your model will give this feature undue predictive influence, which is another problem you want to avoid.

5. Model building

Now we get to the fun part. Model building. You know what you're trying to do and you have the data you want. Now you get to use the data to make predictions. In supervised learning, there are 2 general types of problems: Regression whereby you try to predict a continuous value (e.g. price of a home) and classification whereby you try to predict class association (e.g. Survived vs Not Survived for the Titanic example). There are different algorithms you can use for each type of problem, but generally speaking most people use Linear Regression for regression problems. For classification problems, there are a number of different algorithms including Logistic Regression, k-Nearest Neighbors, Support Vector Machines, Decision Trees, Random Forests and many others. There are a few key steps for model building that I'll go over:

a. Pre-processing

Most, if not all of the time, you'll need to preprocess the data. One step my be feature scaling. Feature scaling ensures that all of your data is on the same scale. Many machine learning algorithms calculate distance between points using Euclidean distance, so this sort of makes it so all features are in the same units. The way I think about it is that if you have one feature measured in meters and another measured in kilometers, you don't want your algorithm to think that two points that are 5 meters away from one another have the same distance between them as points that are 5 kilometers away from one another. Scaling the features ensures proper relationship between features. Also incorporated in preprocessing is regularization. In Linear and Logistic Regression, coefficients for features can get very high which can lead to overfitting and regularization limits the size of the coefficients by implementing a penalty for coefficients. There are 2 types of regularization Ridge (L2) and Lasso (L1).

b. Train-test-split

Earlier, I talked about making sure that your model is generalizable and a way to measure how generalizable your model is is by doing a train-test split. In this step, you take all of your observations and you randomly split them into a training set and a testing set. You use the training set of observations to actually build and fit your model and the testing set to evaluate how well your model performs on data that it has never seen before.

c. Cross-validation

This step is meant to correct for overfitting. Again, overfitting is a problem in which the model is too sensitive to the data that was used to build it, so there is a lot of error (due to variance) when trying to predict targets for new data. This can be contrasted to underfitting in which not enough data is used to build the model so there is error (due to bias). You want to minimize both types of error. When performing cross-validation, the training set is split into k disjoint folds. Then, k different models are built, and each model is built using data from the other k-1 folds. Then the k different models are averaged and your overall model is the averaged model.

Model Testing

Now you can make your predictions! This step is where your testing set comes in. Take that cross-validated model that you built on your training set and use it to predict your target in your testing set. This part is pretty cool because you get an instant response to see how well you did because you know what the actual values for your target are supposed to be and you can compare your actual values to your predicted values.

Model Evaluation

So you made your predictions. Now you want to know how well your model performed. Well, there are a lot of different metrics you can use to evaluate your model. For Linear Regression, the two most common metrics are Mean Squared Error (MSE) and the R$^2$ score. MSE squares the distance between the true values and the predicted values and divides by the number of values. The smaller the MSE the better, although there is no real metric to say what a good MSE is vs a bad one. R$^2$ is a measure of how much better your model is as compared to just having found the average of the true values. If your model is worse than the average, you get a negative R$^2$; however, R$^2$ usually falls between 0 and 1, where 1 means you have a perfect model. An R$^2$ score of .9 is really good, .8 is good, and so on.

For classification models there are a lot of different ways to measure performance and usually many of them are used together. The most basic measurement is accuracy: number of correct predictions / total number of values. There are also confusion matrices which calculate True Positives, False Positives, True Negatives and False Negatives in binary classifications. Confusion matrices also exist for multi-class classification models, but they are are a little harder (more confusing?) to interpret. There are also classification reports, ROC curve, Precision-Recall curves and others and it's usually best to take a look at most or maybe even all of these reports to see how well your model performed.

Improving Your Model

So now you've gone through the process, but you're not done yet. Nobody builds the best model the first time around. Now you go back and do it again, but change a few things. Data science is really about understanding exactly how models work and knowing what to do to improve performance. Maybe one algorithm works better than others for certain types of problems. Why not try a few different ones and compare? Maybe you need to fine-tune some of the parameters like the penalty for your regularization or the number of folds you want when cross-validating. You should also probably rerun your model several times on different train-test splits and see if they vary a lot. After cleaning, this probably takes the most time because there's so much you can do in terms of model tuning that it's easy to get lost in trying to find the perfect set of parameters. But once you find a model you're happy with, it's time to present.

Communicating Results

Finally, the last step in a data science project (and really in any project) is to communicate your results. This is tricky. Unless you're in an academic setting, no one really cares about how you did things, they just want to know how it impacts them. Your client doesn't really care how you were able to fill in those missing values, or that you chose to use a SVM over Logistic Regression. Your client wants to know that you have a good model that will help him/her do what he/she needs to do. So when communicating results, you're really focusing on the performance of your model and how it impacts the client. If you built a model to detect cancer, focus on how much it improves upon what oncologists are currently using to detect cancer. Explain how well your model works and why the client should use it. Don't focus on the math and stats; but make sure that if people do want to know and understand the math and stats behind why you did what you did, you can answer both at a high-level and at a technical level.
